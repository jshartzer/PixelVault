PixelVault — Main Library: scalability notes (Apr 2026 update)
==============================================================
Original review described `ShowLibraryBrowser` + `LibraryVirtualization.cs` before many perf slices landed.

**Implemented since this review:** folder sort-key caching, debounced search, batch cold-index reads, detail render off UI thread, startup cache gating, capture-grid viewport work cancellation, folder-side row recycling (`RecycleVisibleRowElements`), Library browser host extraction (`LibraryBrowserHost`), async/metadata/cover service seams. See `PERFORMANCE_TODO.md` landed summary and `CHANGELOG.md`.

HOW IT IS BUILT (still directionally true)
----------------------------------------
- Left: custom row virtualizer with viewport + overscan.
- Right: capture grid uses virtualization-friendly patterns; off-screen thumbnail/video work is dropped when items leave the viewport (not an unbounded “queue everything forever” model).


REMAINING GAPS (optional / profile-driven)
------------------------------------------
- If traces show **N-limited** memory on huge folders: consider a **full** virtualizing ItemsControl or recycler for every wrap/date-group layout.
- Video-heavy folders: optional poster-first tiles, stricter unload, concurrency caps.
- Folder side: recycling is on for `tileRows`; further pooling is optional.


BOTTOM LINE
-----------
The highest-risk “unbounded capture grid + UI-thread cold index” issues from the original review are **addressed**. Treat this file as **context**, not a task list — use `PERFORMANCE_TODO.md` for open work.
