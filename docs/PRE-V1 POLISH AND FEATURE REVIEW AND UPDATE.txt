PHASE 3 — DETAILED IMPLEMENTATION REQUIREMENTS

Implement the plan according to the requirements below.

--------------------------------------------------
1. DESIGN TOKENS / VISUAL CONSISTENCY
--------------------------------------------------

Goal:
Create a lightweight design system so the app stops feeling like a collection of individually styled controls.

Requirements:
- Inventory all repeated hardcoded colors, corner radii, spacing values, button styles, and text sizes.
- Consolidate them into a small shared design-token layer or equivalent centralized constants/helpers appropriate for this codebase.
- Introduce semantic tokens where practical, such as:
  - background tiers
  - panel/background surfaces
  - border strengths
  - text primary / secondary / muted
  - accent / success / warning / error / info
  - primary / secondary / quiet / destructive action styles
  - corner radius tiers
  - spacing tiers
- Do not over-engineer. Keep it simple and repo-appropriate.
- Replace inconsistent button chrome and inconsistent spacing where possible.
- Ensure toolbar buttons, pills, tiles, banners, and dialogs feel related.
- Normalize hover, pressed, disabled, and selected states.
- Create a clear visual hierarchy for:
  - app-level primary actions
  - contextual secondary actions
  - utility actions
  - destructive actions

Deliverables:
- shared token/style layer
- migrated highest-visibility UI surfaces
- list of remaining legacy areas if not all can be migrated safely

Acceptance criteria:
- fewer repeated style literals
- visible improvement in consistency
- no major visual regressions
- no broken workflows

--------------------------------------------------
2. LOADING STATES / PERCEIVED PERFORMANCE
--------------------------------------------------

Goal:
Make the app feel faster and more trustworthy even when work is still happening.

Requirements:
- Find current blank/awkward loading moments in:
  - library folder grid
  - selected folder detail pane
  - cover art loading
  - metadata fetches
  - ID resolution
  - intake preview / import-related surfaces
  - editor opening flows
- Replace blank waiting periods with localized loading states.
- Add skeletons or placeholder shells where practical:
  - folder tiles
  - hero/cover area
  - screenshot rows
  - metadata panels
- Prefer localized progress indicators instead of a generic full-screen waiting state when only part of the UI is updating.
- Distinguish between:
  - loading
  - empty
  - error
  - stale-but-visible content being refreshed
- Preserve already-visible content when refreshing if possible instead of flashing to empty.
- Where a long-running task is already in progress, show status that tells the user what is happening in plain language.

Acceptance criteria:
- fewer “blank pop-in” moments
- faster perceived response on folder switching and data refresh
- clearer user trust in what the app is doing

--------------------------------------------------
3. INLINE FEEDBACK / REDUCE MODAL OVERUSE
--------------------------------------------------

Goal:
Make feedback feel modern and contextual instead of interruptive.

Requirements:
- Audit all MessageBox/modal uses and classify them:
  - truly blocking
  - confirmation
  - warning
  - recoverable failure
  - success/notice
- Replace non-blocking successes and mild warnings with:
  - toast
  - inline banner
  - message bar near the affected control/section
- Keep modal dialogs only where they are genuinely justified:
  - destructive confirmation
  - critical failure
  - decisions that must be made before continuing
- Add retry affordances where appropriate.
- For operations like cover fetch, ID lookup, setup/config issues, missing tools, or validation issues, prefer inline explanations with action buttons when possible.
- Make sure feedback appears near the relevant surface whenever possible.

Acceptance criteria:
- fewer disruptive dialogs
- clearer recovery paths
- feedback feels attached to the relevant UI

--------------------------------------------------
4. EMPTY STATES
--------------------------------------------------

Goal:
Make all “nothing here” states feel intentional and helpful.

Requirements:
Identify and improve empty states for areas such as:
- no library selected / unavailable library
- no search results
- no upload items waiting
- no cover found
- no starred items
- no smart-view items
- no filename rule matches / no samples
- no IDs available
- no photos in a folder/group
- missing setup/tooling/config

For each empty state:
- provide a short clear title
- one concise explanatory sentence
- one obvious next action when appropriate
- avoid walls of text
- avoid looking broken or unfinished

Acceptance criteria:
- no major app surface should feel like a dead blank area
- empty states guide next action clearly

--------------------------------------------------
5. SIDE PANELS / DRAWERS FOR COMMON EDITS
--------------------------------------------------

Goal:
Reduce unnecessary window sprawl and make common edits feel connected to the main library surface.

Requirements:
- Audit existing editor windows and decide which should stay separate windows vs which should become contextual drawers/panels.
- Prioritize converting only the high-frequency, low-complexity edit flows, for example:
  - quick metadata edit
  - quick ID edit
  - maybe folder actions/details
- Do NOT force every complex editor into a drawer.
- Keep heavyweight table/editor workflows as separate windows if that still makes sense.
- New drawer/panel UX should:
  - preserve context with the selected item/folder visible in the background
  - support save/cancel clearly
  - support loading/error states cleanly
  - avoid fragile state handling
- If full drawer conversion is too risky, create a staged architecture that enables it cleanly.

Acceptance criteria:
- quick edits feel more integrated
- reduced window explosion for common tasks
- no loss of functionality

--------------------------------------------------
6. SUBTLE MOTION / TRANSITIONS
--------------------------------------------------

Goal:
Add restrained, functional motion that clarifies changes without slowing the app down.

Requirements:
- Add motion only where it improves comprehension:
  - tile hover feedback
  - expand/collapse transitions
  - detail pane transition between selections
  - fade-in for newly loaded imagery
  - success affordance for toggles like star/favorite
- Keep motion short and subtle.
- Avoid flashy or decorative animation.
- Avoid anything that makes the app feel slower.
- Respect reduced-motion settings if practical in this environment.
- Make sure motion is consistent with the design token layer.

Acceptance criteria:
- UI feels smoother and more modern
- transitions clarify change
- no annoying animation delays

--------------------------------------------------

--------------------------------------------------

Make the app clearer, more usable, and more trustworthy.

- destructive action clarity
- loading language
- error message wording

- improve confusing microcopy
- ensure settings/setup failures are understandable

Acceptance criteria:
- stronger focus states
- better readability
- clearer error and recovery language
- fewer ambiguous actions
