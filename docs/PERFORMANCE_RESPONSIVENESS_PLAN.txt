PixelVault Native — UI responsiveness and performance plan (reference only)
==========================================================================
Derived from a review of PixelVault.Native (library load, virtualization,
intake preview, thumbnails). Older scalability notes: `LIBRARY_UI_SCALABILITY_NOTES.txt`.

1) Library load / refresh
-------------------------
**Was:** Heavy synchronous index backfill on UI thread for large libraries.  
**Now:** Rebuild/backfill patterns use background work + marshaling; batch tag reads for cold paths. Keep this pattern when changing folder cache or index load.

2) Capture grid
---------------
**Was:** StackPanel queued append; every tile stayed in the tree.  
**Now:** Viewport-driven construction; off-screen work canceled. Optional future: full virtualizing panel if profiling requires it.

3) Sort + search
-----------------
**Was:** Repeated filesystem walks for “Recently Added”; search on every keystroke.  
**Now:** Cached sort keys / capture ticks; debounced committed search term.

4) Thumbnails
--------------
QueueImageLoad + thread-pool + dispatcher remains appropriate; LRU / pipeline split in `LibraryThumbnailPipeline` + `LibraryImageLoadCoordinator`.

5) Intake preview
-----------------
Build summary off UI thread; marshal results with `BeginInvoke` — keep for similar workflows.

6) Threading
-------------
Game-capture keyword inclusion: **landed** — cached mirror on `MainWindow`, no hot-path `Dispatcher.Invoke` (see `PERFORMANCE_TODO.md` 5a).
