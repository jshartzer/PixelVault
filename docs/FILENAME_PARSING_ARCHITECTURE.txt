PixelVault - Filename parsing: extraction, conventions, and user rules
========================================================================

Context (current codebase)
--------------------------
Filename and capture-time logic is split across `PixelVault.Native.cs` and
`MetadataHelpers.cs`, with `GuessSteamAppIdFromFileName` duplicated in
`LibraryFolderIndexing.cs`.

- Game title from basename: `GetGameNameFromFileName` (regex on
  `game_timestamp` shapes, etc.).
- Platform / "review vs manual" routing: `DetectPlatformTags` +
  `CanUpdateMetadata` + `PrimaryPlatformLabel` / `FilenameGuessLabel`.
- Capture time: `ParseCaptureDate` (Steam/Xbox/clip patterns on the filename)
  plus `ParseSteamManualExportCaptureDate` for the bare `yyyyMMddHHmmss` Steam
  export case used in manual import.

Pulling this into one place is justified: one home for rules, one result type,
and no duplicate Steam App ID logic.


Recommended shape: a small "filename intelligence" layer
--------------------------------------------------------

1) Single entry point, rich result

Introduce something like `FilenameParseResult` (or split into
`FilenameClassification` + `FilenameDateHints`) produced by one public API:
`ImportFilenameParser.Parse(string fullPath)`.

If a caller only has a basename, it should still call `Parse(...)` and pass the
basename with no directory. That keeps one public entry point and avoids
separate `ParseFileName(...)` behavior drifting over time.

Rule of the road for inputs:
- The parser should accept the full path so future folder-level defaults can
  participate, but regex conventions should run against `Path.GetFileName(...)`
  or `Path.GetFileNameWithoutExtension(...)` unless a rule explicitly says it
  is path-aware.

- Convention id(s) - e.g. `SteamScreenshot`, `SteamManualExport`, `Ps5Share`,
  `XboxClip`, `Unknown`.
- Platform tags - same semantics as today's `DetectPlatformTags`.
- Optional `DateTime?` - best guess from the basename (see below).
- Optional Steam AppID, optional game title hint (from the same rules that
  drive `GetGameNameFromFileName`).
- Confidence / source - e.g. `ExplicitPattern` vs `Heuristic` so the UI can say
  "guessed from filename" vs "user rule".

Call sites (`BuildReviewItems`, `BuildManualMetadataItems`, library indexing,
folder naming) should only consume this result, not own regex.


2) Ordered conventions, not one giant regex block
-------------------------------------------------

Model each known pattern as a small class or delegate: name, priority,
`TryMatch(fileName, out ...)`. Run them in priority order with explicit
conflict rules. Recommended default:

- First winning convention sets the primary convention id and primary platform
  label.
- Steam AppID, title hint, and capture time come from that same winning
  convention unless a later stage explicitly overrides them.
- Tags may be unioned from multiple passes only when they are additive and
  non-conflicting; otherwise the winning convention owns them too.

Example default:
- `PS5` + `PlayStation` is additive and OK.
- `Steam` + `Xbox` from different matches is conflicting and should not be
  merged unless an explicit rule says both are valid for that pattern.

That supports detection based on known filename conventions and is easy to
extend (`Switch 2`, `GeForce Experience`, `OBS`, etc.) without touching WPF.


3) Date/time: one general pass, separate from platform
------------------------------------------------------

Today, date extraction is effectively platform-coupled (patterns live next to
Steam/Xbox tag detection). A cleaner approach:

- Add a `FilenameDateExtractor` (or a second phase inside the same parser) that
  tries generic patterns in a fixed order, for example:
  - `yyyyMMddHHmmss` as a token (Steam manual export already does this).
  - `yyyy-MM-dd` / `yyyy_MM_dd` with optional time segments.
  - ISO-8601-like fragments in the middle of names.
  - Unix ms/us in the basename (`Steam clip_` already).

Then choose one "best" timestamp with simple rules: prefer full date+time over
date-only; prefer anchored tokens (e.g. `_stamp_`) over random 14-digit
substrings; optional sanity window (not year 2099, etc.).

Platform tags can still set behavior (e.g. Xbox "preserve filesystem time")
without owning how the date string is parsed.

DateTime / timezone rule:
- Filename-derived times should be treated as local capture times and stored as
  `DateTimeKind.Local` so they line up with `GetLibraryDate` ordering, UI
  display, and EXIF writes. If a rule parses UTC or epoch-based values, convert
  them to local time before returning the final parse result.
- In .NET terms, `DateTime.TryParseExact(...)` often returns
  `DateTimeKind.Unspecified`, so the implementation should explicitly apply
  `DateTime.SpecifyKind(parsedValue, DateTimeKind.Local)` for local filename
  stamps before they are persisted or mixed with EXIF/tool output.


4) Storing "example filenames" and desired classification
---------------------------------------------------------

Pure "example -> classify" matching is fuzzy and brittle unless you constrain
it. Practical tiers:

A) User rules file (JSON next to app data): regex or glob + fields
   (`PlatformTags`, `Convention`, optional date group names)
   - Pros: precise, versionable, testable
   - Cons: needs a minimal format doc

B) "Rule from example" in UI
   - User pastes `Game_20240315_001.png`; app suggests a regex with capture
     groups they can edit
   - Cons: one-time UI effort

C) Per-source-folder defaults
   - e.g. everything under `D:\Switch\Album` -> tag `Nintendo`
   - Cons: doesn't help mixed folders

Best default: A + optional B. Store rules as data, load at startup, and merge
after built-in conventions by convention id. "Merge" here should mean:

- A custom rule with a new convention id appends to the rule set.
- A custom rule with the same convention id replaces the built-in rule with
  that id for the whole result, not just one field.
- If power-user overrides are needed later, model them as full-rule
  replacement rather than per-field merging.
- If the custom-rule store contains multiple rows with the same convention id,
  do not leave the result order-dependent. Recommended default: reject or
  collapse duplicates on save so runtime load sees at most one custom row per
  convention id.

For PixelVault specifically, prefer the same persistence pattern as the
existing game/photo index runtime store: built-in defaults in code, custom
overrides in the SQLite index database, with optional export/import later if a
portable file format is useful.


5) Tests
--------

Add `FilenameParserTests` (or golden JSON) with one row per basename: expected
tags, date, Steam id, game hint. Include mostly basename-only cases plus a
small set of full-path cases once folder-aware defaults exist, so tests cover
the real public API instead of only the regex core. Fastest way to extend
conventions without regressions.


6) Migration path
-----------------

1. Add `Import/FilenameParsing/` (or `Services/Import/`) with parser + result
   types + built-in rules moved verbatim from `PixelVault.Native.cs` and
   `MetadataHelpers.cs`.
2. Replace `GuessSteamAppIdFromFileName` in `LibraryFolderIndexing.cs` with the
   shared helper.
3. Unify `ParseSteamManualExportCaptureDate` into the general date phase so any
   matching `yyyyMMddHHmmss` basename gets the same treatment, not only the
   manual export branch.
4. Add DB-backed custom filename conventions and unmatched-sample capture once
   the API is stable.


Bottom line
-----------
Treat filename parsing as a data-driven pipeline (ordered convention matchers +
a separate date extraction pass + optional user rules), expose a single result
DTO, and lock behavior with table-driven tests. That gives separation from the
UI, general date detection, and a clear place to plug in "examples" as explicit
rules rather than implicit fuzzy matching.

Default implementation choice:
- Prefer embedded built-in conventions in code plus SQLite-backed custom
  overrides in the same index database the app already uses for runtime state.
- If a portable export/import format is needed later, add it as a secondary
  import-export surface instead of making JSON the primary runtime source.
