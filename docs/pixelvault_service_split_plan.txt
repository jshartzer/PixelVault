PixelVault Service Split Plan (living stub)
==========================================

**Status (Apr 2026):** Phases **1–6** of the original extraction order are **implemented in source** (settings, metadata behind `IMetadataService`, SQLite/index persistence via `IndexPersistenceService` + related types, `ILibraryScanner`, `IImportService`, `ICoverService`, `IFileSystemService`). **Phase 7** (smaller `MainWindow`, composition-root cleanup) continues **incrementally** via `HANDOFF.md`, `PERFORMANCE_MONOLITH_SLICE_PLAN.md`, and `MAINWINDOW_EXTRACTION_ROADMAP.md` (A–F complete; further monolith shrink).

The **full** phase-by-phase task lists, “good prompt” examples, and historical wording are frozen in  
**`docs/archive/pixelvault_service_split_plan_HISTORICAL.txt`**.

Goal (unchanged)
----------------
Break centralized logic into services **without** changing visible behavior. Prefer extraction and wiring over redesign.

Guiding rules
-------------
- Small phases; compile after each step.
- Behavior-preserving unless a change explicitly documents an exception.
- Thin interfaces; avoid giant “Manager” types.
- ExifTool stays behind `IMetadataService` for now.
- No heavy DI requirement — explicit constructors are fine.

Recommended folder layout (reference)
-----------------------------------
`/Models`, `/Services/Config`, `/Services/Metadata`, `/Services/Library`, `/Services/Intake` (import pipeline: `ImportService`, orchestration, headless coordinator, analysis), `/Services/Covers`, persistence under `Services/Indexing` / `Storage`, `/Services/IO`, UI under `/UI`.

Suggested service map (implemented shapes)
----------------------------------------
| Area | Types (typical) |
|------|------------------|
| Settings | `ISettingsService`, `SettingsService`, `AppSettings` |
| Metadata | `IMetadataService`, `MetadataService` |
| Library scan | `ILibraryScanner`, `LibraryScanner`, `ILibraryScanHost` |
| Game index orchestration | `IGameIndexService`, `GameIndexService`, `ILibrarySession` |
| Persistence / photo index | `IIndexPersistenceService`, `IndexPersistenceService` (SQLite `photo_index`, etc.) |
| Import | `IImportService`, `ImportService`, `ImportWorkflow` shell |
| Covers | `ICoverService`, `CoverService` |
| File I/O seam | `IFileSystemService`, `FileSystemService` |
| Logging | Operational logging still mostly via `MainWindow` / app hooks; dedicated `ILogService` optional |

Phase 7 — ongoing (not a single PR)
------------------------------------
- Shrink `PixelVault.Native.cs` and orchestration in partials/hosts.
- Route remaining glue through sessions/services when touching a file (`ARCHITECTURE_REFACTOR_PLAN.md`).
- Sequenced milestones + Phase B (service organization after extraction): `docs/plans/PV-PLN-EXT-002-service-extraction-and-organization.md`.

Cross-cutting (many landed; extend only when touching paths)
--------------------------------------------------------------
- Cancellation on long workflows.
- Async-first for network and cover paths.
- Clear thread boundaries (no background mutation of UI field bags without an owner).

What NOT to do without an explicit decision
-------------------------------------------
- Replace ExifTool in the same pass as structural moves.
- Repo-wide nullable enable **as a requirement** for every slice.
- Full MVVM rewrite as a gate for service work.

Definition of success
---------------------
`MainWindow` trends toward **UI coordination**; library/import/metadata/cover **rules** live in services; the app behaves the same; future batching and storage work can target services instead of the *primary* monolith file.
